Understandable principle

The principle of Understandable is the third guideline of the Web Content Accessibility Guidelines (WCAG) and ensures that both the information and the operation of the user interface are easy to understand. This principle focuses on making digital content and interfaces predictable, readable, and user-friendly, especially for people with cognitive, learning, or language-related disabilities.

Under the Understandable principle, there are three guidelines that help ensure websites and applications are easy to comprehend and use. These guidelines focus on consistent behavior, readable content, and error handling.

  1. Readable - Make text content readable and understandable. This includes using clear language, defining unusual words or abbreviations, and providing the ability to identify the language of the content so that assistive technologies can interpret it correctly.
  2. Predictable - Make web pages appear and operate in predictable ways. Users should be able to anticipate how things will behave. This includes consistent navigation, avoiding unexpected changes when elements receive focus, and ensuring that user-initiated actions don’t cause sudden context changes.
  3. Input Assistance - Help users avoid and correct mistakes. This includes clearly identifying input errors, offering suggestions for correction, and allowing users to review and confirm information before final submission, which is especially important for forms.

Each of these guidelines contains Success Criteria that define measurable requirements at various WCAG levels (A, AA, AAA). These criteria help developers ensure that all users, including those with cognitive or intellectual challenges, can easily understand and interact with digital content.

Guidelines of Success Criteria

There are a total of 17 Success Criteria (SC) in the Principle of Understandable.

  1. Guideline 3.1: Readable

    • 3.1.3 Unusual Words (AAA)
    • 3.1.4 Abbreviations (AAA)
    • 3.1.5 Reading Level (AAA)
    • 3.1.6 Pronunciation (AAA)
  2. Guideline 3.2: Predictable

    • 3.2.2 On Input (A)
    • 3.2.3 Consistent Navigation (AA)

  3. Guideline 3.3: Input Assistance
    • 3.3.1 Error Identification (A)
    • 3.3.2 Labels or Instructions (A)

    • 3.3.4 Error Prevention (Legal, Financial, Data) (AA)

Examples of Guideline 3.1: Readable

Guideline 3.1 of the Web Content Accessibility Guidelines (WCAG) ensures that the text content on a website is easy to read and understandable for as many users as possible. This includes specifying the language of the content and providing support for users who may struggle with reading or comprehension.

Here are examples of how to meet Guideline 3.1:

  1. 3.1.1 Language of Page (A)

    2. The default human language of each web page must be programmatically identified. Example: An English website uses the language tag lang="en" to identify the language of the page.

    • Good Example: The HTML element includes lang="en" to clearly indicate the content is in English.
    • Bad Example: No language attribute is defined, so screen readers may mispronounce the text.
  2. 3.1.2 Language of Parts (AA)

    3. When a passage or phrase of text is in a different language than the main page, it must be marked with the correct language attribute. Example: A blog post in English includes a French quote marked with lang="fr".

    • Good Example: The quote “C’est la vie” is wrapped with so screen readers switch pronunciation correctly.
    • Bad Example: The French phrase is not marked, so it is read incorrectly by assistive technologies.
  3. 3.1.3 Unusual Words (AAA)

    4. A mechanism is provided for understanding unusual or complex words, including jargon, idioms, and abbreviations. Example: An academic site provides definitions or glossaries for technical terms like “photosynthesis” or “mitochondria.”

    • Good Example: Offers tooltips or linked glossary definitions for technical terms.
    • Bad Example: Uses technical vocabulary without explanation, which may confuse non-experts.
  4. 3.1.4 Abbreviations (AAA)

    5. A method must be available to identify the meaning of abbreviations. Example: The abbreviation “WCAG” is accompanied by a definition or expansion on first use: “Web Content Accessibility Guidelines (WCAG).”

    • Good Example: The full term is given on first mention, with a tooltip or parenthetical explanation.
    • Bad Example: The site uses “HTML,” “URL,” and “API” without explaining what they stand for.
  5. 3.1.5 Reading Level (AAA)

    6. If text requires reading skills more advanced than lower secondary education, a simpler version or supplemental content must be provided. Example: A government website uses plain language and offers a simplified summary for legal texts.

    • Good Example: Provides a “Summary in Plain Language” section for each dense document.
    • Bad Example: Uses complex legal or academic language without explanation.
  6. 3.1.6 Pronunciation (AAA)

    7. When the pronunciation of a word is necessary to understand the content, a way to access it must be available. Example: An online dictionary includes an audio button next to each word for pronunciation.

    • Good Example: Words like “colonel” include phonetic spelling or a “play pronunciation” button.
    • Bad Example: Words with unusual pronunciation are shown with no way to hear or understand them.

Guideline 3.2: Predictable

Guideline 3.2 of the Web Content Accessibility Guidelines (WCAG) ensures that websites behave in consistent and expected ways. When content or interface elements behave unexpectedly, it can confuse users — especially those using assistive technologies or those with cognitive disabilities. This guideline aims to reduce surprises by promoting consistent navigation and interactions.

Here are examples of how to meet Guideline 3.2:

  1. 3.2.1 On Focus (A)

    2. When an element receives focus (e.g., by keyboard navigation), it must not initiate a significant change like submitting a form or opening a new window. Example: When a user tabs to a form field, it doesn’t automatically move to another page or change content.

    • Good Example: Tabbing into the field only highlights it; no other changes occur.
    • Bad Example: Tabbing into a dropdown menu causes the page to reload with different content.
  2. 3.2.2 On Input (A)

    3. Changing a form control’s value (like selecting an option) must not automatically trigger a major action unless the user is informed beforehand. Example: A dropdown selection changes a page section, but only after the user clicks a submit button.

    • Good Example: The user selects an item and then clicks “Go” to confirm the change.
    • Bad Example: Selecting an item in a dropdown instantly redirects the user.
  3. 3.2.3 Consistent Navigation (AA)

    4. Navigation links and menus that appear on multiple pages must be presented in the same order each time. Example: A website's top menu includes “Home, About, Services, Contact” in that order on every page.

    • Good Example: The layout and order of navigation elements remain consistent site-wide.
    • Bad Example: Navigation order changes randomly on different pages.
  4. 3.2.4 Consistent Identification (AA)

    5. Elements with the same function must be identified consistently across the site. Example: A “search” feature always uses the same magnifying glass icon and label, regardless of the page.

    • Good Example: All search fields are labeled as “Search” and use the same icon and placement.
    • Bad Example: A search bar is labeled “Find” on one page and “Quick Lookup” on another.
  5. 3.2.5 Change on Request (AAA)

    6. Changes of context (like loading new content or moving to another page) should only occur at the user's request. Example: A multi-step form doesn’t proceed to the next step until the user clicks “Next.”

    • Good Example: The user must click a clearly labeled button to proceed.
    • Bad Example: The page jumps to the next step automatically when the user finishes typing.

Examples of Guideline 3.3: Input Assistance

Guideline 3.3 of the Web Content Accessibility Guidelines (WCAG) focuses on helping users avoid and correct mistakes when interacting with web forms or other input mechanisms. It is especially helpful for people with cognitive, language, and motor disabilities.

Here are examples of how to meet Guideline 3.3:

  1. 3.3.1 Error Identification (A)

    2. If an input error is detected, it must be identified and described to the user in text. Example: A form shows an error message saying, “Email address is required” when the field is left empty.

    • Good Example: The error message is clearly stated in text: “Please enter a valid email address.”
    • Bad Example: The field just turns red without explaining the issue.
  2. 3.3.2 Labels or Instructions (A)

    3. Labels or instructions must be provided when user input is required. Example: A password field includes a label and a note: “Password (8+ characters, at least one number).”

    • Good Example: Each field is clearly labeled and includes helpful guidance.
    • Bad Example: Fields are shown without labels or hints, causing confusion.
  3. 3.3.3 Error Suggestion (AA)

    4. When an error is detected and suggestions for correction are known, they should be provided. Example: An error message for a form states: “Username already exists. Try adding numbers or choosing a different name.”

    • Good Example: It offers a suggestion: “The name 'john123' is taken. Try ‘john456’ instead.”
    • Bad Example: It just says, “Error.”
  4. 3.3.4 Error Prevention (Legal, Financial, Data) (AA)

    5. For web pages that involve legal or financial transactions, users must be able to review and correct their input before final submission. Example: An e-commerce checkout process includes a confirmation screen showing billing and shipping details before purchase.

    • Good Example: The user is shown a summary and must click “Confirm Order” after review.
    • Bad Example: Clicking “Submit” instantly places the order without review.
  5. 3.3.5 Help (AAA)

    6. Context-sensitive help is available for more complex input tasks. Example: A job application form includes a “Help” icon next to fields with additional instructions when needed.

    • Good Example: Each tricky section includes a help icon or tooltip explaining what’s expected.
    • Bad Example: The form has no guidance or help, leading to mistakes.
  6. 3.3.6 Error Prevention (All) (AAA)

    7. All user input is either reversible, checked for errors, or confirmed by the user before final submission. Example: A web app saving sensitive data displays a summary and asks for confirmation before saving.

    • Good Example: A confirmation dialog appears before finalizing the action, and there’s an “Undo” option.
    • Bad Example: Submitting a form immediately deletes data with no confirmation or undo option.
