Robust

The principle of Robust is the fourth and final principle of the Web Content Accessibility Guidelines (WCAG). It ensures that content can be interpreted reliably by a wide variety of user agents, including assistive technologies. This principle focuses on building content that is technically sound, follows standard coding practices, and remains accessible as technology evolves.

Under the Robust principle, there is one main guideline that ensures digital content is developed using proper coding techniques so that it remains usable across different platforms, browsers, and assistive tools like screen readers.

  1. Compatible - Maximize compatibility with current and future user agents, including assistive technologies. This includes writing clean and valid code, using proper semantics, labeling elements correctly, and ensuring that web technologies (HTML, ARIA, etc.) are implemented in a way that supports accessibility tools effectively.

The guideline includes Success Criteria that require developers to follow standards for coding and ensure that user input components are identifiable and accessible programmatically. This is crucial for ensuring long-term accessibility, especially as browsers and assistive technologies update over time.

Guidelines of Success Criteria

There are a total of 3 Success Criteria (SC) under the Robust principle.

  1. Guideline 4.1: Compatible
    • 4.1.1 Parsing (A)
    • 4.1.2 Name, Role, Value (A)
    • 4.1.3 Status Messages (AA)

With the Robust principle, developers ensure that web content works seamlessly with both current and emerging technologies, which is vital for long-term accessibility and compatibility.

Examples of Guideline 4.1: Compatible

Guideline 4.1 of the Web Content Accessibility Guidelines (WCAG) ensures that content is robust enough to be reliably interpreted by a wide variety of user agents, including assistive technologies. This principle helps ensure future compatibility as technologies evolve.

Here are examples of how to meet Guideline 4.1:

  1. 4.1.1 Parsing (A)

    2. Ensure that HTML or XML content is properly coded so that it can be reliably parsed by browsers and assistive technologies. Example: A webpage uses properly nested HTML tags and closes all elements correctly.

    • Good Example: All tags are properly opened and closed. Elements are nested in a logical order.
    • Bad Example: A
      tag is opened but never closed, causing screen readers to misinterpret the page layout.
  2. 4.1.2 Name, Role, Value (A)

    3. User interface components must have appropriate names, roles, and values that can be programmatically determined and used by assistive technologies. Example: A custom checkbox component correctly exposes its role (checkbox), state (checked/unchecked), and name (label text) to screen readers.

    • Good Example: The button is coded with ARIA roles and labels so assistive technology identifies it correctly as a button.
    • Bad Example: A custom-built button visually looks clickable but doesn’t announce itself as a button to a screen reader.
  3. 4.1.3 Status Messages (AA)

    4. Status messages (such as errors or success confirmations) must be programmatically exposed to assistive technologies without receiving focus. Example: A message like “Form submitted successfully” is announced by the screen reader when it appears on the screen.

    • Good Example: The message is coded with aria-live="polite" so screen readers read it aloud automatically.
    • Bad Example: A confirmation message appears visually but is not announced by the screen reader.
