The principle of Perceivable is the first guideline of the Web Content Accessibility Guidelines (WCAG) and ensures that all users can perceive the content of a website or application. This principle focuses on making information and user interface components available to the senses of sight, sound, and touch.

Under the Perceivable principle, there are four guidelines that help ensure digital content is accessible to all users, including those with disabilities. These guidelines are.

  1. Text Alternatives – Provide text alternatives for non-text content, such as images, audio, and video, so that assistive technologies can present the information in an accessible format.
  2. Time-Based Media – Offer alternatives for time-based media, such as captions for videos and transcripts for audio content, to support users who are deaf or hard of hearing.
  3. Adaptable – Create content that can be presented in different ways without losing information or structure, such as ensuring proper heading structures and flexible layouts.
  4. Distinguishable – Ensure content is easy to see and hear, using sufficient color contrast, resizable text, and the ability to adjust audio settings.

Each of these guidelines includes several Success Criteria that provide measurable requirements for compliance at different WCAG levels (A, AA, AAA). These criteria help developers and designers create web content that is accessible to people with visual, auditory, and other sensory impairments.

Examples of guideline 1.1

Guideline 1.1 of the Web Content Accessibility Guidelines (WCAG) focuses on Providing Text Alternatives for non-text content to ensure accessibility for users with disabilities, including visually impaired individuals who rely on screen readers.

There are six main types of images as per WCAG Guideline 1.1: Providing Text Alternatives. Each type requires different handling for accessibility.

  1. Informative Images

    2. Informative images convey important content that enhances understanding. These images provide visual details that add meaning to the surrounding text. For example, a photograph of Albert Einstein in a biography article serves to show what he looked like at a particular time. A good text alternative should describe the image’s purpose, such as "Albert Einstein in 1921, wearing a suit and bow tie." A bad example would be using a vague alt text like "Einstein" or leaving it empty, as this does not provide meaningful information to visually impaired users.

  2. Decorative Images

    3. Decorative images are purely aesthetic and do not add meaning to the content. They are often used for visual appeal, such as background patterns, borders, or ornamental graphics. Since they do not convey any useful information, their alt attribute should be left empty, ensuring that screen readers skip them. A bad practice would be adding unnecessary alt text like "Fancy border image," which could be distracting for screen reader users. Instead, simply setting the alt attribute to empty, such as alt="", allows assistive technologies to ignore the image.

  3. Functional Images

    4. Functional images serve an interactive purpose, such as acting as buttons, links, or controls. For example, a search icon should have an alt text like "Search" so that users know its function. If the alt text is left empty or is too vague, such as "Icon," users may not understand what action the image performs. Providing clear and concise descriptions ensures that users relying on assistive technology can navigate the website effectively.

  4. Complex Images (Graphs, Charts, Diagrams, Maps)

    5. Complex images convey data and require a detailed description. These include charts, infographics, and diagrams that present information visually. For example, a bar chart showing population growth should have an alt text like "Bar chart showing population growth from 2010 to 2020, increasing by 30%." A poor example would be simply stating "Chart" or "Graph," as it does not provide the necessary context. When possible, a long description or a textual data summary should also be provided for better accessibility.

  5. Text Images

    6. Text images contain written content embedded within an image. Since screen readers cannot interpret text inside images, an accurate alternative text should be provided. For instance, an image of a motivational quote should have an alt text that includes the full quote, such as "The only limit to our realization of tomorrow is our doubts of today. - Franklin D. Roosevelt." A bad example would be using a generic alt text like "Quote image," which does not convey the message contained within the image. If possible, using actual text instead of an image is recommended for better accessibility.

  6. Image CAPTCHAs

    7. Image CAPTCHAs are used to verify whether a user is human, often by displaying distorted letters or numbers. These pose a challenge for visually impaired users, as screen readers cannot interpret them. A poor example would be using an alt text like "Captcha image," which does not help the user complete the task. The best practice is to provide an alternative method, such as an audio CAPTCHA or a text-based challenge, to ensure accessibility for all users.

Examples of guideline 1.2

Guideline 1.2 of the Web Content Accessibility Guidelines (WCAG) focuses on providing alternatives for time-based media, such as audio and video content, so that users with disabilities, including those who are visually or hearing impaired, can access the information.

  1. 1.2.1 Audio-Only and Video-Only - Prerecorded (AA)

    2. A podcast episode discusses financial planning tips. Since there is no visual content, a full transcript is provided on the website for users who are deaf or prefer reading.

    • Good Example: A downloadable text transcript is available that includes the full conversation and speaker names.
    • Bad Example: The podcast has no transcript, preventing accessibility for people who cannot hear.
  2. 1.2.2 Captions - Prerecorded (A)

    3. A university uploads an online lecture video where the professor explains key concepts. To make it accessible for deaf or hard-of-hearing students, closed captions are provided.

    • Good Example: The video includes synchronized captions that display the spoken words and relevant sounds (e.g., "[Applause]" or "[Laughter]").
    • Bad Example: The video has no captions, making it inaccessible to users with hearing impairments.
  3. 1.2.3 Audio Description or Media Alternative - Prerecorded (A)

    4. A documentary features visuals of landscapes and historical buildings, with background music but no spoken narration. To assist visually impaired users, an audio description is added, describing key visual elements.

    • Good Example: An alternative version of the video includes an audio description that narrates what is happening on screen.
    • Bad Example: The documentary plays music over visuals without explaining the scenes to blind users.
  4. 1.2.4 Captions - Live (AA)

    5. A company hosts a live webinar about new business trends. To accommodate deaf or hard-of-hearing participants, real-time captions are provided.

    • Good Example: A live captioning service transcribes the speaker’s words in real time.
    • Bad Example: The webinar is broadcast without captions, making it inaccessible.
  5. 1.2.5 Audio Description (Prerecorded) (AA)

    6. A university records a science experiment demonstration where a professor mixes two chemicals, causing a color change and smoke to appear. Since this is important information for understanding the video, an audio description is added.

    • Good Example: An audio description is provided, The professor pours a clear liquid into a beaker. The solution slowly turns blue, then begins to produce thick white smoke.
    • Bad Example: The professor mixes the chemicals silently, and there is no explanation of what happens.
  6. 1.2.6 Sign Language - Prerecorded (AA)

    7. A government website provides a video tutorial explaining how to file taxes online. To assist users who use sign language, a sign language interpreter is shown in the corner of the video.

    • Good Example: A sign language interpreter is visible on-screen, making the content more accessible.
    • Bad Example: The video only has voice narration and no sign language support.
  7. 1.2.7 Extended Audio Description (Prerecorded) (AAA)

    8. A history documentary about ancient Egyptian artifacts includes scenes with detailed carvings and inscriptions that are crucial to understanding the video. However, the narrator is speaking continuously, leaving no time to insert audio descriptions.

    • Good Example: The video pauses at key moments, and an extended audio description explains: The camera zooms in on a stone tablet with intricate hieroglyphics. The carvings depict a pharaoh standing before the sun god Ra, offering a golden scepter.
    • Bad Example: No additional description is provided, so blind users miss out on the visual details.
  8. 1.2.8 Media Alternative (Prerecorded) (AAA)

    9. A business training video explains how to manage customer complaints through role-playing scenarios. Some users may not be able to watch or hear the video, so a full media alternative is required.

    • Bad Example: The website only provides captions, with no text transcript of actions or visuals.
    • Good Example: A full script is available that includes: All dialogue, Descriptions of character actions, Key visual cues, Background noises.
  9. 1.2.9 Audio-only (Live) (AAA)

    10. A company hosts a live audio-only investor call, where executives discuss quarterly earnings.

    • Good Example: A real-time text transcript is available, updated as speakers talk. This could be done via: Live captioning services and A stenographer providing real-time transcription.
    • Bad Example: The company provides no transcript, leaving deaf or hard-of-hearing users unable to follow.

Examples of Guideline 1.3: Adaptable

Guideline 1.3 ensures that content is structured in a way that allows assistive technologies (such as screen readers) to interpret and present the information properly. This means using proper headings, labels, tables, and semantic HTML so that content remains accessible and adaptable to different devices and user needs.

  1. 1.3.1 Info and Relationships (A)

    2. Information that is visually structured (such as tables, headings, and lists) must, Example: A university website displays a course schedule.

    • Good Example: The schedule is structured properly, with clear headings like "Course Name," "Instructor," and "Time," so screen readers can read it in a logical order.
    • Bad Example: The schedule is presented as plain text without clear labels, making it difficult for screen reader users to understand which information belongs to which column.
  2. 1.3.2 Meaningful Sequence (A)

    3. The content must be presented in a logical reading order that makes sense when Example: A news website displays a headline, an image, and a paragraph of text.

    • Good Example: The headline appears first, followed by the image, and then the paragraph, ensuring the content is understood as intended.
    • Bad Example: The paragraph appears first in the reading order before the headline, making it confusing for users who rely on screen readers.
  3. 1.3.3 Sensory Characteristics (A)

    4. Instructions should not rely only on visual cues such as color, shape, or position because not all users can perceive them. Example: A website provides instructions for navigating a form by saying, "Click the red button to continue.

    • Good Example: The button includes both text and a label, such as "Click the button labeled ‘Continue’," so users can identify it even if they can't see the color.
    • Bad Example: The button is only identified by its color, making it inaccessible to colorblind users.
  4. 1.3.4 Orientation (A)

    5. Content must be usable in both portrait and landscape modes. Example: A mobile banking app is designed to work only in landscape mode.

    • Good Example: The app works in both portrait and landscape modes, adapting to the user's preference.
    • Bad Example: Users who cannot rotate their devices, such as those using a wheelchair-mounted tablet, cannot access the app properly.
  5. 1.3.5 Identify Input Purpose (AA)

    6. Form fields should be designed so that assistive technologies can recognize their purpose and help users fill them out. Example: An online shopping website asks for the user's name, email, and address.

    • Good Example: The form allows autofill and clearly labels each input field (such as "Full Name," "Email Address," and "Shipping Address"), so screen readers and password managers can assist users.
    • Bad Example: The form does not provide hints or allow autofill, making it harder for users with disabilities to complete it.
  6. 1.3.6 Identify Purpose (AAA)

    7. Icons, buttons, and interactive elements should have clear descriptions so assistive technologies can understand their function. Example: A music player app has buttons for play, pause, and volume, represented only by symbols.

    • Good Example: Each button has a clear label, such as "Play," "Pause," and "Increase Volume," so assistive technologies can describe them properly.
    • Bad Example: The buttons have no text labels, making it difficult for screen reader users to know what each button does.

Guideline 1.4: Distinguishable

Guideline 1.4 ensures that content is easy to see and hear by separating foreground from background, improving contrast, and allowing users to adjust text and multimedia presentation.

  1. 1.4.1 Use of Color (A)

    2. Color should not be the only way of conveying meaning, as users with color blindness or low vision may not perceive it correctly. Example: A form highlights errors using only red text.

    • Good Example: The error message is accompanied by an icon and text, such as "Error: Please enter a valid email address," making it accessible to all users.
    • Bad Example: The error message is displayed in red without any additional indication. A colorblind user may not notice the issue.
  2. 1.4.2 Audio Control (A)

    3. Users should be able to pause, stop, or adjust the volume of audio that plays for more than 3 seconds. Example: A website automatically plays background music.

    • Good Example: The website provides a clear button to pause or mute the audio, allowing users to control their experience.
      • Bad Example: The audio plays without an option to pause, making it difficult for users to focus on reading the content.
  3. 1.4.3 Contrast (Minimum) (AA)

    4. Text and images of text should have sufficient contrast against their background for readability. Example: A news article displays black text on a dark gray background.

    • Good Example: The text is displayed in black on a white background, ensuring clear visibility.
    • Bad Example: The contrast is too low, making it hard to read for users with visual impairments.
  4. 1.4.4 Resize Text (AA)

    5. Users should be able to enlarge text up to 200% without losing functionality or readability. Example: A website uses small text that cannot be resized.

    • Good Example: The website’s text scales properly when zoomed in, ensuring readability without breaking the layout.
    • Bad Example: The text becomes cut off or overlaps when users zoom in, making it unreadable.
  5. 1.4.5 Images of Text (AA)

    6. Text should be displayed as real text, not embedded in images, so it can be resized and read by assistive technologies. Example: A poster with event details is shared as an image.

    • Good Example: The event details are provided in actual text alongside the image so that all users can read them.
    • Bad Example: The event information is in an image without alternative text, making it inaccessible to screen reader users.
  6. 1.4.6 Contrast (Enhanced) (AAA)

    7. Text should have an even higher contrast ratio for users with low vision. Example: A website has light gray text on a white background.

    • Good Example: The text is dark gray or black on a white background, improving readability.
    • Bad Example: The contrast is too weak, making it hard for visually impaired users to read.
  7. 1.4.7 Low or No Background Audio (AAA)

    8. Audio should be clear, with minimal background noise, to help users with hearing impairments. Example: A video tutorial has loud background music.

    • Good Example: The video ensures clear narration with minimal background noise.
    • Bad Example: The background music overpowers the speaker’s voice, making it difficult to understand.
  8. 1.4.8 Visual Presentation (AAA)

    9. Users should be able to customize text appearance, such as font size, color, and spacing, for better readability. Example: A website has fixed text styles and spacing.

    • Good Example: The site allows users to adjust font size and spacing for a more comfortable reading experience.
    • Bad Example: The text is cramped and cannot be adjusted, making it difficult to read.
  9. 1.4.9 Images of Text (No Exception) (AAA)

    10. Text should never be embedded in images unless essential. Example: A menu is displayed as an image instead of real text.

    • Good Example: The menu is provided in real text, ensuring accessibility.
    • Bad Example: Users cannot copy, resize, or have the menu read aloud by a screen reader.
  10. 1.4.10 Reflow (AAA)

    11. Content must adapt to different screen sizes without requiring users to scroll in multiple directions (both horizontally and vertically). Example: A blog page is designed with a fixed width.

    • Good Example: The content reflows properly, adjusting to fit smaller screens without horizontal scrolling, improving accessibility for all users.
    • Bad Example: When viewed on a mobile device, the user has to scroll horizontally to read each line, making it difficult for those with limited mobility.
  11. 1.4.11 Non-Text Contrast (AAA)

    12. UI components (such as buttons, icons, and form controls) must have enough contrast to be easily distinguishable. Example: A submit button on a form is a light gray color against a slightly darker gray background.

    • Good Example: The button has a high contrast color (such as dark blue on a white background), ensuring it stands out clearly.
    • Bad Example: The contrast is too low, making it hard for users with low vision to locate the button.
  12. 1.4.12 Text Spacing (AAA)

    13. Users must be able to increase text spacing (line height, letter spacing, word spacing, and paragraph spacing) without losing functionality or readability. Example: A long article on a website has tightly packed text.

    • Good Example: The site allows for text spacing adjustments, ensuring no loss of readability when spacing is increased.
    • Bad Example: When users adjust the text spacing using browser settings or custom stylesheets, words overlap, making it unreadable.
  13. 1.4.13 Content on Hover or Focus (AAA)

    14. When additional content (such as tooltips, pop-ups, or dropdown menus) appears on hover or focus, users should be able to easily dismiss it and still access the underlying content. Example: A shopping website displays product details in a tooltip when a user hovers over a product name.

    • Good Example: The tooltip remains visible until the user moves focus away or presses the "Esc" key, giving them time to read the content.
    • Bad Example: The tooltip disappears immediately when the cursor moves, making it hard for users with motor impairments to read the information.
